How this app talks to the VISUAL database

what, where, and how it stays safe

Source: the API Toolkit references in “References/Visual PDF Files/Text Conversion/”. Summarized for this app’s interactions.

Connection and authentication

  • Opens a short-lived client connection using Lsa.Data.Dbms.OpenLocal (username/password) or OpenLocalSSO (Windows SSO). Connection info comes from Database.config.
  • Database: the operational VISUAL database you registered (SQL Server or Oracle), commonly the VMFG schema.
  • Safety: the API owns the connection/credentials; on user switch or error the connection is closed with Dbms.Close. No direct SQL writes through this path.

Read-only lookups used by the UI

  • Part validation: Lsa.Vmfg.Inventory.Part Exists/Find/Load to verify a part ID and read details.
  • Location picker: Lsa.Vmfg.Inventory.Location Browse/Load; availability by location via Lsa.Shared.GeneralQuery (parameterized, read-only).
  • Work order checks: Lsa.Vmfg.ShopFloor.WorkOrder Exists/Find and GetWorkOrderSummary for status and remaining quantities.
  • Safety: results are disconnected DataSets; writes occur only via controlled Save on business objects.

Inventory Transfer

  • What happens: build one row in Lsa.Vmfg.Inventory.InventoryTransaction via Prepare ? NewInputRow ? Save with TRANSACTION_TYPE = TRANSFER, setting at minimum ENTRY_NO, PART_ID, SITE_ID, FROM_*, TO_*, and QTY.
  • Databases: VISUAL inventory; app MAMP DB updated separately for client-side audit/history.
  • Safety: Toolkit enforces rules and wraps in a single DB transaction with rollback on validation failures.

Work Order Issue/Receipt

  • Issue: TRANSACTION_TYPE = ISSUE; supply WO keys (WORKORDER_TYPE/BASE_ID/LOT_ID/SPLIT_ID/SUB_ID) and OPERATION_SEQ_NO when required, plus FROM_* and QTY.
  • Receipt: TRANSACTION_TYPE = RECEIPT; supply target warehouse/location and same WO keys; TRACE sub-table included when required.
  • Safety: same transactional guarantees; validation rejects invalid rows and rolls back the unit of work.

Other operations

  • ADJUST_IN/ADJUST_OUT and interbranch via IbtShipment/IbtReceipt exist but are not primary in this app.

Central safety mechanisms

  • Disconnected data; Transactional saves with commit/rollback; Auto-numbering without races; Read-only GeneralQuery; Scope/permissions by VISUAL identity; Short-lived licensing.

Databases in play

  • VISUAL (SQL Server/Oracle; VMFG schema): business reads/writes via Toolkit.
  • MAMP (app DB): app-owned settings, local history, and exception/reporting tables. Never modified through VISUAL APIs.
